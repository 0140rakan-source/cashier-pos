# دليل تطبيق نظام الإضافات (Product Add-ons / Modifiers)

هذا الدليل يشرح التعديلات المطلوبة لإضافة نظام **مجموعات الخيارات** للمنتجات في مشروع `cashier-pos`.
تم بناء التعديلات بحيث **لا تكسر** أي بيانات أو ميزات حالية، ومتوافقة بالكامل مع **Windows + SQLite + Electron**.

> ⚠️ ملاحظة مهمة عن قاعدة البيانات: مشروعك يستخدم **SQLite** عبر `prisma db push`
> (ملف `prisma/migrations/` القديم خاص بـ PostgreSQL وقديم/غير صالح، و`migration_lock.toml`
> ما زال مكتوباً فيه `postgresql`). لذلك استخدم **`db push` وليس `migrate dev`** — التفاصيل بالأسفل.

---

## 1) الملفات التي تم تعديلها أو إضافتها

| الملف | النوع | الوصف |
|------|------|-------|
| `backend/prisma/schema.prisma` | **تعديل** | إضافة 3 جداول جديدة + حقول علاقة على `Product` و`SaleItem`. الجداول القديمة لم تُمس. |
| `backend/src/modules/products/modifier-groups.routes.js` | **جديد** | API لإدارة مجموعات الخيارات وخياراتها (GET / PUT حفظ كامل / DELETE). |
| `backend/src/app.js` | **تعديل** | سطر `require` + سطر `app.use('/api/modifier-groups', ...)`. |
| `backend/src/modules/sales/sales.routes.js` | **تعديل** | إعادة كتابة دالة `createSale`: تحقق من المجموعات (إجباري/min/max) في السيرفر + حفظ snapshot للإضافات داخل الطلب. |
| `frontend/src/pages/Products.jsx` | **تعديل** | استبدال محرّر الإضافات المسطّح القديم بمحرّر **مجموعات وخيارات**. |
| `frontend/src/pages/POS.jsx` | **تعديل** | نافذة اختيار الإضافات أصبحت تعتمد المجموعات (radio/checkbox)، مع تحقق إجباري وسعر نهائي حي، وإرسال `optionId` للسيرفر. |

> الملفات القديمة `modifiers.routes.js` و`ProductModifier`/`SaleItemModifier` **بقيت كما هي**
> حتى لا تتعطل أي بيانات قديمة؛ النظام الجديد يعمل بجانبها.

### كيفية التطبيق
انسخ كل ملف من مجلد `out/` فوق نظيره في المشروع (نفس المسارات). الملفات الجديدة (`modifier-groups.routes.js`) تُضاف فقط.
استخدم `git diff` للمراجعة قبل الـ commit. لا ترفع `node_modules` أو `.env` أو `data/cashier.db` أو ملفات `.exe`.

## 2) الجداول الجديدة في قاعدة البيانات

تمت إضافة 3 جداول (مع `@@map` لأسماء الجداول الفعلية):

### `product_modifier_groups` — مجموعات الخيارات
| الحقل | النوع | الوصف |
|------|------|------|
| `id` | String | المعرّف |
| `productId` | String | المنتج (Cascade عند الحذف) |
| `nameAr` / `nameEn` | String | اسم المجموعة |
| `required` | Boolean | هل الاختيار إجباري |
| `minSelect` | Int | أقل عدد اختيارات |
| `maxSelect` | Int | أعلى عدد اختيارات (=1 يعني radio، >1 يعني checkbox) |
| `sortOrder` | Int | ترتيب العرض |
| `isActive` | Boolean | مفعّلة |

### `product_modifier_options` — الخيارات داخل المجموعة
| الحقل | النوع | الوصف |
|------|------|------|
| `id` | String | المعرّف |
| `groupId` | String | المجموعة (Cascade) |
| `nameAr` / `nameEn` | String | اسم الخيار |
| `priceDelta` | Decimal | السعر الإضافي (قد يكون 0) |
| `isAvailable` | Boolean | متاح أو لا |
| `sortOrder` | Int | ترتيب العرض |

### `order_item_modifiers` — snapshot الإضافات داخل الطلب
يخزّن نسخة ثابتة وقت البيع، تبقى صحيحة حتى لو تغيّر المنتج أو حُذف الخيار لاحقاً:
| الحقل | النوع | الوصف |
|------|------|------|
| `id` | String | المعرّف |
| `saleItemId` | String | عنصر الفاتورة (Cascade) |
| `optionId` | String? | مرجع للخيار (يصبح null لو حُذف الخيار — SetNull) |
| `groupName` | String | اسم المجموعة وقت البيع (snapshot) |
| `optionName` | String | اسم الخيار وقت البيع (snapshot) |
| `priceDelta` | Decimal | السعر الإضافي وقت البيع |
| `quantity` | Int | الكمية (افتراضي 1) |

## 3) كيفية تشغيل تحديث قاعدة البيانات

> مهم: **لا تستخدم `prisma migrate dev`** — سيفشل لأن مجلد الـ migrations الحالي PostgreSQL.
> استخدم `db push` الذي يطابق ما استُخدم لإنشاء قاعدة البيانات الحالية.

على جهازك (Mac أو Windows)، من داخل مجلد `backend`:

```bash
cd backend

# 1) خذ نسخة احتياطية من قاعدة البيانات أولاً (مهم)
#    Mac/Linux:
cp ../data/cashier.db ../data/cashier_backup.db
#    Windows:
#    copy ..\data\cashier.db ..\data\cashier_backup.db

# 2) ادفع المخطط الجديد إلى قاعدة البيانات (ينشئ الجداول الجديدة فقط)
npx prisma db push

# 3) أعد توليد Prisma Client
npx prisma generate
```

`db push` آمن هنا: سيُنشئ الجداول الجديدة الثلاثة فقط، ولن يحذف أي بيانات موجودة
(لا توجد تغييرات على الأعمدة القديمة، فقط إضافات).

لو ظهرت رسالة تطلب reset (لا يجب أن تظهر مع الإضافات فقط)، **أوقف** ولا توافق على الـ reset، وراجع المخطط.

## 4) كيف تختبر إضافة منتج مع إضافات

### أ) إعداد الإضافات (شاشة المنتجات)
1. شغّل المشروع: `npm run dev` من الجذر.
2. ادخل **المنتجات** → اضغط أيقونة الترس (⚙️ الإضافات) بجانب منتج (مثلاً "برجر").
3. اضغط **"إضافة مجموعة خيارات"** وأنشئ:
   - مجموعة **"اختيار الشطة"**: إجباري ✓، الأدنى = 1، الأعلى = 1 → ستعمل كـ **radio**.
     - خيارات: "بدون شطة" (0)، "شطة عادية" (0)، "شطة زيادة" (0).
   - مجموعة **"إضافات"**: اختياري، الأدنى = 0، الأعلى = 5 → ستعمل كـ **checkbox**.
     - خيارات: "إضافة جبن" (2)، "إضافة بطاطس" (5)، "إضافة صوص" (1).
4. اضغط **"حفظ الخيارات"**.

### ب) البيع (نقطة البيع POS)
1. ادخل **نقطة البيع**، اختر "برجر".
2. ستظهر نافذة الإضافات:
   - "اختيار الشطة" بأزرار دائرية (radio) — لا يمكن المتابعة دون اختيار (إجباري).
   - "إضافات" بمربعات (checkbox) — حتى 5 اختيارات.
3. اختر "شطة زيادة" + "إضافة جبن" → السعر النهائي يظهر **20.00 ر.س** (18 + 2).
4. اضغط **"إضافة للسلة"**. في السلة يظهر تحت اسم المنتج:
   - ✓ شطة زيادة
   - ✓ إضافة جبن +2.00
5. أكمل الدفع. في الإيصال وفاتورة المطبخ ستظهر الإضافات أيضاً.

### ج) التحقق من الـ snapshot
بعد إتمام بيع فيه إضافات، ارجع لشاشة المنتج واحذف خيار "إضافة جبن" واحفظ.
الطلب القديم في السجلات يبقى يعرض "إضافة جبن +2" لأن النظام يخزّن نسخة ثابتة (snapshot).

### نقاط التحقق السريعة
- [ ] منتج بدون مجموعات خيارات → يُضاف للسلة مباشرة بدون نافذة (سلوك قديم محفوظ).
- [ ] مجموعة إجبارية بدون اختيار → زر "إضافة للسلة" معطّل + رسالة "إجباري".
- [ ] مجموعة `max=1` → اختيار واحد فقط (radio).
- [ ] مجموعة `max>1` → اختيارات متعددة حتى الحد الأقصى.
- [ ] السعر النهائي يتحدث حياً قبل الإضافة للسلة.
- [ ] الإجمالي في الفاتورة صحيح (شامل الضريبة 15%).

## 5) كيف تبني نسخة Windows لاحقاً بدون مشاكل

التعديلات **لا تمس** أي شيء خاص بـ Windows:
- لم تُضف أي مكتبة جديدة (لا في backend ولا frontend) — فقط كود JS/JSX.
- لم تتغير ملفات `.bat` ولا `electron/` ولا إعدادات `build` في `package.json`.
- لا توجد مسارات خاصة بـ Mac.

خطوات البناء كما هي في `WINDOWS-BUILD-HANDOFF.md`، باختصار على جهاز Windows:

```bat
:: 1) ثبّت الاعتماديات (مرة واحدة)
setup-windows.bat

:: 2) حدّث قاعدة البيانات بالمخطط الجديد قبل البناء
cd backend
npx prisma db push
npx prisma generate
cd ..

:: 3) ابنِ المثبّت
build-installer.bat
```

> تذكير: تأكد أن `backend/.env` على جهاز Windows فيه:
> `DATABASE_URL="file:../../data/cashier.db"`
> (نفس ما هو موجود في `.env.example`). الـ schema الآن provider = sqlite، وهو المطلوب لـ Windows.

## 6) ملخص منطق الأمان (مهم)

- **السعر لا يُؤخذ من العميل أبداً**: السيرفر يجلب `salePrice` و`priceDelta` من قاعدة البيانات ويعيد الحساب.
- **التحقق من القواعد في السيرفر**: `required` / `minSelect` / `maxSelect` تُفحص في `createSale` حتى لو تلاعب أحد بالواجهة.
- **snapshot الطلب ثابت**: حذف أو تعديل خيار لاحقاً لا يغيّر الطلبات السابقة.
- جميع التغييرات تمت داخل `prisma.$transaction` للحفاظ على تماسك البيانات والمخزون.

## 7) تم اختباره

تم التحقق من المنطق فعلياً عبر محاكاة على قاعدة بيانات SQLite حقيقية:
- برجر 18 + جبن (+2) + شطة زيادة (0) = **20.00** للوحدة، الإجمالي مع الضريبة **23.00**، الباقي صحيح.
- رفض المجموعة الإجبارية عند عدم الاختيار ✓
- رفض تجاوز الحد الأقصى ✓
- بقاء الـ snapshot بعد حذف الخيار ✓
- حفظ/تعديل/حذف المجموعات والخيارات (مزامنة كاملة) ✓
- جميع ملفات الـ backend تجتاز `node --check`، وملفات الـ frontend تُترجم عبر esbuild بنجاح.
